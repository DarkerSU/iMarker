Koa+TS+Swagger 实现一个Node服务 ​

搭建过程 ​

  1. 项目初始化和安装依赖同上

  2. 初始化 TypeScript 配置

执行如下命令来生成tsconfig.json

bash
npx tsc --init

将tsconfig.json文件中的target选项改为es2017，表示我们要使用较新的 ECMAScript 版本。

json
{
  "compilerOptions": {
    "target": "es2017",    // 修改 target 的值
    "module": "commonjs",
    "outDir": "./dist",
    "strict": true,
    "esModuleInterop": true,
    "sourceMap": true
  }
}
  1. 创建项目结构

我们希望项目结构更加清晰，能够更好的管理路由、控制器等业务代码，所以创建如下目录：

txt
- src/
  - app.ts  // 应用程序入口
  - routes/  // 存储路由文件
    - index.ts  // 定义路由
  - controllers/  // 存储控制器文件
    - index.ts  // 处理请求和响应的逻辑
  1. 实现应用程序入口

我们需要的实现应用程序的入口，这里将定义一个名为 app 的常量来保存应用程序实例。还将把路由分离出去，使用koa-router插件辅助路由。

执行以下命令安装插件：

bash
npm install koa koa-router

代码如下：

typescript
// app.ts
import Koa, { Context, Next } from 'koa';
import Router from 'koa-router';

import routes from './routes';

// 创建Koa实例
const app = new Koa();

// 路由
routes.forEach((route) => {
  const router = new Router();
  router[route.method](route.path, async (ctx: Context, next: Next) => {
    // 执行路由处理函数
    await route.action(ctx, next);
  });
  app.use(router.routes());
});

// 监听
app.listen(3000, () => {
  console.log('Server started on port 3000');
});

上面代码中，我们使用 forEach 循环遍历了所有路由，在路由中使用koa-router注册该路由和路由处理函数的处理逻辑。

  1. 实现路由和控制器

在routes目录下，实现一个简单的路由文件index.ts：

typescript
import { Context, Next } from 'koa';

export default [
  {
    path: '/hello',
    method: 'get',
    action: async (ctx: Context, next: Next) => {
      ctx.body = 'hello koa';
    },
  },
];

同时，在controllers目录下，创建一个中间件处理的逻辑文件index.ts:

typescript
import { Context, Next } from 'koa';

export default class MainController {
  /**
   * GET /
   * 返回欢迎语
   */
  static async sayHello(ctx: Context, next: Next) {
    ctx.body = 'hello koa and typescript!';
  }
}

然后在routes目录的index.ts中添加一项路由：

typescript
import { Context, Next } from 'koa';
import MainController from '../controllers';

export default [
  {
    path: '/',
    method: 'get',
    action: MainController.sayHello,
  },
  {
    path: '/hello',
    method: 'get',
    action: async (ctx: Context, next: Next) => {
      ctx.body = 'hello koa';
    },
  },
];

这里分别添加了两个路由，一个是/路径仅仅返回一个欢迎词，另一个是/hello路径向浏览器输出字符串'hello koa'。

  1. 启动项目

在命令行中执行以下命令启动该项目：

bash
npm run watch-ts

这个命令启动了一个监视程序来监听 TypeScript 文件的变化并重新生成编译文件，同时也启动了我们的 Koa

  1. 处理错误

在 Koa 中，有一个或多个中间件可以处理错误，也称错误中间件。在我们的应用程序中添加一个简单的错误中间件以处理服务器错误。

首先，修改我们的app.ts文件，引入 Koa 中内置的error事件处理器，用于处理未知错误：

typescript
import Koa, { Context, Next } from 'koa';
import Router from 'koa-router';

import routes from './routes';

const app = new Koa();

app.on('error', (err, ctx) => {
  console.error('Server error', err);
});

// 路由
...

app.listen(3000, () => {
  console.log('Server started on port 3000');
});

上面代码中，我们使用app.on来监听error事件，以便在调用堆栈中没有其他有效错误处理程序的情况下处理错误。

然后，添加一个自定义错误中间件，用于处理已知错误：

typescript
// app.ts

import Koa, { Context, Next } from 'koa';
import Router from 'koa-router';

import routes from './routes';

const app = new Koa();

app.use(async (ctx: Context, next: Next) => {
  try {
    await next();
  } catch (err) {
    ctx.status = err.statusCode || err.status || 500;
    ctx.body = {
      message: err.message,
    };
    ctx.app.emit('error', err, ctx);
  }
});

// 路由
...

app.listen(3000, () => {
  console.log('Server started on port 3000');
});

上面代码中，我们使用try...catch语句捕获路由函数所抛出的错误，并在捕获错误后使用ctx.status将状态码设置为错误的状态码，使用ctx.body返回错误信息。除此之外，还可以使用ctx.app.emit('error', ...)方法将错误抛出到应用程序的error事件中，以便让其上级错误中间件或回调函数处理该错误。

  1. 日志记录

在开发和生产环境下都需要记录日志。这样，我们可以通过分析日志了解应用程序的行为、异常和性能。

下面是一个可用性很强的日志记录中间件：

typescript
// app.ts

import Koa, { Context, Next } from 'koa';
import Router from 'koa-router';

import routes from './routes';

const app = new Koa();

// 日志记录
app.use(async (ctx: Context, next: Next) => {
  const start = Date.now();
  await next();
  const millisecond = Date.now() - start;
  console.log(`${ctx.method} ${ctx.url} ${ctx.status} - ${millisecond}ms`);
});

// 错误处理
...

// 路由
...

app.listen(3000, () => {
  console.log('Server started on port 3000');
});

在这个中间件中，我们使用了Date.now函数记录请求响应时间，并将这个时间输出到控制台日志上。注意到这个中间件段在路由前面，以便按照请求调用路由函数。

  1. 验证请求参数

当我们处理请求时，通常需要验证请求参数，以确保其符合规范和逻辑。使用koa及其插件都可以使我们的验证任务变得容易。

例如，我们就可以使用koa-json-schema-validator插件来验证 POST 请求的 JSON 数据中是否包含所需的参数。

安装koa-json-schema-validator：

bash
npm install koa-json-schema-validator

然后，在app.ts中使用它：

typescript
import Koa, { Context, Next } from 'koa';
import Router from 'koa-router';
import validate from 'koa-json-schema-validator';

import routes from './routes';

const app = new Koa();

// 验证请求参数
app.use(validate({}));

// 日志记录
...

// 错误处理
...

// 路由
...

app.listen(3000, () => {
  console.log('Server started on port 3000');
});

这个validate()中间件可接受一个 JSON 模式作为验证规则。例如，我们可以为 POST 请求添加以下 JSON 模式：

typescript
{
  type: 'object',
  properties: {
    name: {type: 'string'},
    age: {type: 'number'},
  },
  required: ['name', 'age'],
}

然后，在路由函数中使用这个规则：

typescript
// controllers/index.ts

import { Context, Next } from 'koa';
import joi from 'joi'; // 引入验证插件

export default class MainController {
  /**
   * POST /
   * 创建用户
   */
  static async createUser(ctx: Context, next: Next) {
    const schema = joi.object({
      name: joi.string().required(),
      age: joi.number().required(),
    });
    // 验证请求参数
    const {value, error} = schema.validate(ctx.request.body);
    if (error) {
      ctx.throw(400, error);
    }
    ctx.body = {
      message: 'success',
    };
  }
}

这里，我们使用 Joi，一个流行的验证插件，创建一个 JSON 模式。这个模式验证POST请求的JSON数据中是否包含name和age参数，如果不符合，则通过ctx.throw抛出错误。

以上是一个比较完整的基于Koa和TypeScript的服务器实现的示例，可以使用此示例作为参考，将其用于实现其他类型的服务器，如REST API服务器等等。

集成Swagger ​

添加 Swagger 的文档是为 API 添加在线文档和交互式 API 界面的好方法。通过 Swagger，我们可以轻松地为 API 的样式创建在线文档，并获得 Swagger 提供的 API 调试界面。在这里，我们使用 OpenAPI（先前称为 Swagger）规范来描述我们的 API 和自动生成在线文档界面。

在 Koa 中添加 Swagger 的过程并不复杂，我们只需要使用一个称为 Swagger UI 的 UI 库来显示文档，并使用一个称为 Swagger JSDoc 的库来将注释转换为 OpenAPI 规范。

在继续下面的步骤之前，请确保安装了如下依赖库：

bash
npm install swagger-ui koa2-swagger-ui koa2-swagger-jsdoc
  1. 首先，我们需要在代码中定义一些 OpenAPI 文档。在这里，我们将创建一个名为 swagger.ts 文件，其中我们将定义我们的 API 中数据类型的模式，并使用 "openapi" 版本 OpenAPI 规范。
typescript
// swagger.ts

import { IDocumentOptions } from 'koa2-swagger-ui'; 

const documentOptions: IDocumentOptions = {
  definition: {
    openapi: '3.0.0',
    info: {
      title: 'My API',
      version: '1.0.0',
      description: '描述我的 API'
    },
  },
  apis: ['./controllers/*.ts'], // 这里指定了我们的 API 层，注意要使用 TypeScript 文件扩展名 ".ts"
};

export default documentOptions;

在上面的代码中，我们定义了一个名称为 documentOptions 的文档配置对象，该对象包含我们应用程序的基本信息和应用程序的模式和路径等信息。

  1. 接下来，我们需要在每个控制器函数中添加 swagger 相关的注释，以便让 Swagger-JSDoc 描述我们的 API。下面的示例代码演示了如何描述我们的示例 createUser API：
typescript
import { Context, Next } from 'koa';
import Joi from 'joi';
import {request, summary, body, responses, tags} from 'koa-swagger-decorator';

@tags(['User'])
export default class UserController {
  @request('post', '/users')
  @summary('创建用户')
  @responses({
    200: {description: '成功'},
    400: {description: '错误的请求参数'}
  })
  @body({
    name: {type: 'string', required: true},
    age: {type: 'number', required: true},
  })
  static async createUser(ctx: Context, next: Next) {
    const schema = Joi.object({
      name: Joi.string().required(),
      age: Joi.number().required(),
    });

    const {value, error} = schema.validate(ctx.request.body);
    if (error) {
      ctx.throw(400, error);
    }

    ctx.body = {
      message: 'success',
    };
  }
}

在上面的代码中，我们使用 Koa Swagger Decorator 中提供的 @request、@summary、@body 和 @responses 声明了我们的 createUser API。这些注释告诉 Swagger-JSDoc 如何生成文档，包括请求类型、请求路径、请求主体、响应状态和响应说明等.

  1. 修改app.ts文件，实现 Swagger 的功能。在上面导入的 koa2-swagger-* 来引用 Swagger UI 和 Swagger-JSDoc 库：
typescript
// app.ts

import Koa, { Context, Next } from 'koa';
import Router from 'koa-router';
import swaggerUi from 'koa2-swagger-ui';
import {
  useSwagger,
  useKoaSwaggerUI,
} from 'koa2-swagger-jsdoc';

import routes from './routes';
import documentOptions from './swagger';

const app = new Koa();

// Swagger-JSDoc 中间件
useSwagger(app, documentOptions);

// Swagger-UI 中间件
useKoaSwaggerUI(app, {
  routePrefix: '/docs',   // URL 前缀
  swaggerOptions: {
    url: '/swagger.json',  // 指定 Swagger JSON 文档
  },
});

// 日志记录

在上面的代码中，我们声明了两个中间件：useSwagger() 和 useKoaSwaggerUI() 。

useSwagger() 中间件将 Swagger-JSDoc 组件中定义的文档配置对象解析为一个 OpenAPI 3 兼容的 JSON 文档，并将这个文档放在 '/swagger.json' 路径中，以便 Swagger UI 从此处读取文档并显示在 Swagger UI 界面中。

useKoaSwaggerUI() 中间件将 Swagger UI 界面注册到应用程序中，并设置UI界面的URL前缀。并使用 url 选项指定了需要从哪里获取 SwaggerJSON 文档，注意这里要与 useSwagger() 中间件中的路径保持一致。

  1. 启动应用程序运行：
bash
npm start

此时应该可以访问 http://localhost:3000/docs 查看生成的 Swagger 文档了，在这个页面中，你可以看到所有的 API 接口，以及它们的说明、请求参数和响应说明等等。还可以使用交互式控制台来调试和测试 API。

如果您被 Swagger 的功能深深吸引，可以尝试在上面展示的基础之上继续完善 Swagger UI，并进一步提高 API 的可读性和可测试性。

集成Clog ​

在上面我们已经提到了如何在 Koa 中添加日志记录中间件。在通过 Swagger UI 调试 API 的时候，有时需要记录 API 的请求和响应数据来调试问题。这就需要在日志记录中间件中添加更多的字段来记录请求和响应数据。下面是一个通过使用 Colog（另一个插件）来在日志记录中间件中增加更多字段的例子：

  1. 首先，安装Colog：
bash
npm install colog
  1. 修改app.ts，并将 Colog 日志记录中间件代码添加到“日志记录”段中：
typescript
// app.ts
import Koa, { Context, Next } from 'koa';
import Router from 'koa-router';
import swaggerUi from 'koa2-swagger-ui';
import {
  useSwagger,
  useKoaSwaggerUI,
} from 'koa2-swagger-jsdoc';
import colog from 'colog';

import routes from './routes';
import documentOptions from './swagger';

const app = new Koa();

// Swagger-JSDoc 中间件
useSwagger(app, documentOptions);

// Swagger-UI 中间件
useKoaSwaggerUI(app, {
  routePrefix: '/docs',
  swaggerOptions: {
    url: '/swagger.json',
  },
});

// 日志记录
app.use(async (ctx: Context, next: Next) => {
  const start = Date.now();
  await next();
  const millisecond = Date.now() - start;
  colog.info({
    method: ctx.method,
    url: ctx.url,
    status: ctx.status,
    request: ctx.request.body,
    response: ctx.response.body,
    time: `${millisecond}ms`,
  });
});

// 错误处理

// 路由
...

app.listen(3000, () => {
  console.log('Server started on port 3000');
});

在上面的代码中，我们使用 Colog 来增强我们的日志记录。在中间件中，我们记录请求方法、请求地址、状态码、请求和响应体、请求响应时间等等。

  1. 启动应用程序运行：
bash
npm start

现在，当您调用一个 API 时，您将在控制台中看到类似于以下内容的输出：

txt
{"method":"POST","url":"/users","status":200,"request":{"name":"John Doe","age":30},"response":{"message":"success"},"time":"5ms"}

改日志可以使你方便的查看请求和响应数据，方便开发和调试。

注意：在将生产代码放在生产环境中时，应该将日志记录保持到一个更适合生产环境的日志设置中，例如滚动文件日志或云日志记录。
